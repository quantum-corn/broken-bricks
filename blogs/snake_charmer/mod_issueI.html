Modules

Think of modules like plug-in features.
The basic language that we have learned so far is only the basic structures required to make a programming language. Think of a simple task that doesn't come built-in with the language, like picking a number out of a set or a range of values. It's common enough for a lot of people to need but not an elementary structural unit of the language. So someone decides to code it. Now everyone writing the code is a waste of time and effort. So the person who wrote the code decides to make their code available to the public. Now everyone can reuse the code. But copy pasting is inefficient in code, as we can all agree, so to easen up this process of reusing already written code, the concept of module was created. Think of plugging in the code wherever you need and being able use the features of it seamlessly.

Accessing the Module

Every module has a name.
This might seem obvious but just like variables it is a case-sensitive, unique name, so we need to be exact about it. The random module is called random.
To use a module, the original source code of it must be there on the machine on which the code is to run.
Again it might seem obvious, but quite often while we try to run our code on a friend's machine, it doesn't work, and we waste time trying to fix the code only realize later that, it was a matter of module availability. random come with Python by default, but others you need to install manually. Installation instruction can almost always be found at the module's website and documentation, or just plain Google it up. Most often we use something called package managers to install and manage pacakges which contain modules (for the time being, just keep it simple package = modules). Python has it's own package manager called pip. I will probably write a separate blog on pip, so I will no digress into it here. Now that we have our module let's see how to use it in our code. The first step is importing it in our code, which just a nerdy way of saying, 'Let's plug it in!''
(Advanced)
Another thing to take care of, is that the module source code is at a location where Python is going to look for it. Usually this means the locations on the $PATH global variable of the operating system. This usually isn't a worry because modules installed via pip, which is where we get most of our modules, go to a location which Python stores on $PATH when Python is installed. This becomes a thing to look out for when we are playing around with our own module or something we got from somewhere other than a package manager, like pip. One way to simplify such situations is to use the full path of the module source code when we import the module into our code.
Importing is done by the keyword import. At this point I must digress a little and talk about the structure of modules. Just how files are structured inside a directory, with subdirectories, modules may have submodules, and you can import only submodules instead of the whole module, using keyword from, or the dot operator, which is a matter of object orientation and needs not be worried about at this stage. This is what you should do as often as possible to keep your code slim and trim. It's just more efficient. Also while importing a module we can give it an alias using keyword as, which is just a nerdy way of saying nickname. import random
import random as ran
from matplotlib import pylot
from matplotlib import pyplot as plt
import matplotlib.pyplot
import matplotlib.pyplot as plt The first case os importing module called random. The second case does the same but also gives it the alias (nickname) ran (we will see later why this matters). The third and fifth cases imports a submodule pyplot from a module matplotlib which is a module to create plots and graphs using different ways. The fourth and sixth cases, as you can guess, do the same but also give the submodule a nickname plt.

Using Modules

I'll start with a simple idea. Any module can only provide 2 kinds of features to your code.

  • Data
  • Functions
We need to know the exact variable name in which the data want to access is stored inside the module, or the name of the function we want to use in our code. This is part that involves learning how to use a certain module. And it is not hard. And as always Google is always there to help. Let us look at some examples.
>>> import random
>>> rng=random.Random()
>>> # This line was technically a class instance creation, as the Random() is used. And it's an established tradition to start constructor names with capital letters. Constructors are special functions used to create copies of classes. But this is a matter of Object Orientation.
>>> # For now just think of it as a box from which yo ucan pull out random numbers.
>>> rng.randrange(1,7)
2
>>> rng.randrange(1,7)
5
>>> rng.random()
0.47788220195260545
>>> rng.random()
0.6594531447371549
>>> import math
>>> math.pi
3.141592653589793
>>> math.e
2.718281828459045
>>> math.sqrt(4.0)
2.0
Notice how sometimes parenthesis is used and sometimes not. We know we use parenthesis for functions. So this explain how sometimes we are using functions to get our job done eg. rng.random(), and sometimes, we use plain data, eg. math.pi.
Notice how the module name is used before every use of a data or function of that module. This is a way to tell the system, for example, 'get the value of the variable pi from the source code of the math module', or 'use a function called sqrt() the definition of which is in the source code of the math module'. But then another question that presents itself is why was rng.random() used instead of the module name. The answer again is Object Orientation. That one line where we created a copy of the class random, via Random(). SO now instead of accessing the original source code we are accessing the copy that we named rng. These copies are called objects. It is one of the many magical things that Object Orientation lets us do. Now let us look at some examples using aliases.
>>> import math as m
>>> math.sin(math.pi)
Traceback (most recent call last):
File "<stdin>", line 1, in <module>
NameError: name 'math' is not defined
>>> m.sin(m.pi)
1.2246467991473532e-16
Also do notice well that creating object and aliasing a module are not the same thing deep down.
Obviously, I haven't even scratched the surface of the capabilities of the modules I used. But my point here was to demonstrate the use of use of modules, the modules themselves can be explored in depth in some other session.

Special cases

Now let us look at 2 special scenarios.

  • We can import all the subelements of a module by using the * shortcut, eg. from math import *
    >>> from math import *
    >>> math.pi
    Traceback (most recent call last):
    File "<stdin>", line 1, in <module>
    NameError: name 'math' is not defined
    >>> pi
    3.141592653589793

    Notice that when imported using the from ... import ... structure the subelements are imported individually and therefore can not be addressed via the module. Even though this might seem like good idea initially as the typing is considerably reduced, it is actually not a smart thing to do, because all the variables and functions from the module have entered you global scope and therefore can cause unpredicatable and unmanageable interference with functions and variables that you actually create in your global scope. If the idea of scope and namespace seems unfamiliar, do not worry. Just go learn about it, I'll prolly have a session on that. After than come here and read this again. I hope it will make more sense.

  • With some previous information about scope and namespace, it is of interest to know that modules can be imported within function scopes. These imports will not be accessible outside the scope of that function, that is to say, in some other function or in the global scope.
    >>> def root(a):
    ...   import math
    ...   return math.sqrt(a)
    ...  
    >>> math.sqrt(5)
    Traceback (most recent call last):
    File "<stdin>", line 1, in <module>
    NameError: name 'math' is not defined
    >>> root(5)
    2.23606797749979
    >>> math.sqrt(5)
    Traceback (most recent call last):
    File "<stdin>", line 1, in <module>
    NameError: name 'math' is not defined
     Notice how the module was not available in the global scope. And even if you think it was the fuction was just defined and not executed, notice that it was unavailable even after the function was executed. This is neat little trick to play with namespaces/scopes.
